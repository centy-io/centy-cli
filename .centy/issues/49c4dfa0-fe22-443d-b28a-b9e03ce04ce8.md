# Consolidate CLI get commands to use generic GetItem RPC

Replace the type-specific `get issue`, `get doc`, `get asset`, `get org`, `get user` commands with a single generic `centy get <type> <id>` command that calls the daemon's `getItem` gRPC endpoint.

## Current state

- `src/commands/get/issue.ts` calls `daemonGetIssuesByUuid` / `fetchAndDisplayIssue` (type-specific)
- `src/commands/get/doc.ts` calls `daemonGetDocsBySlug` (type-specific)
- `src/commands/get/asset.ts`, `get/org.ts`, `get/user.ts` — each with their own daemon wrappers
- The daemon already has a generic `GetItem` RPC in `grpc-client-items.ts` that accepts `itemType` and `itemId`

## Target UX

```bash
centy get issue 1
centy get issue abc123-uuid
centy get doc my-doc-slug
centy get epic 1
centy get bug abc123-uuid --json
centy get issue abc123-uuid --global
```

## Implementation

### New files

- `src/commands/get/item.ts` — Generic oclif command with `<type>` and `<id>` as positional args. Flags: `--json`, `--global`, `--project`

### Modified files

- `src/commands/get/issue.ts` — Replace with alias/wrapper delegating to generic command
- `src/commands/get/doc.ts` — Same
- `src/commands/get/asset.ts` — Same
- `src/commands/get/org.ts` — Same (if org/user are treated as item types)
- `src/commands/get/user.ts` — Same

### Notes

- The `--global` flag (cross-project UUID search) should be preserved for item types that support it
- Output formatting should adapt based on the item type's config (display number, status, priority, etc.)
- `get project` is a different entity and may stay separate
