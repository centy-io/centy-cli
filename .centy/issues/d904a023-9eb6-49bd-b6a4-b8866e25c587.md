# Consolidate CLI create commands to use generic CreateItem RPC

Replace the type-specific `create issue` and `create doc` commands with a single generic `centy create <type>` command that calls the daemon's `createItem` gRPC endpoint. This enables creating items for any registered item type (issues, docs, epics, or any custom type) without needing a dedicated CLI command per type.

## Current state

- `src/commands/create/issue.ts` calls `daemonCreateIssue` (type-specific RPC)
- `src/commands/create/doc.ts` calls `daemonCreateDoc` (type-specific RPC)
- The daemon already has a generic `CreateItem` RPC in `grpc-client-items.ts` that accepts an `itemType` string field (e.g., "issues", "docs", "epics")
- The generic `CreateItemRequest` is defined in `generated/centy/v1/generic_item.ts`

## Target UX

```bash
centy create issue --title "Bug in login" --priority high
centy create doc --title "Getting Started" --body "# Guide"
centy create epic --title "Auth overhaul" --status open --priority 1
centy create bug --title "Crash on startup" --status open
```

The `<type>` argument is the singular name of any registered item type. The command resolves the plural form and config from the daemon.

## Implementation

### New files

- `src/commands/create/item.ts` — Generic oclif command with `<type>` as a positional arg. Flags: `--title`, `--body`/`--description`, `--status`, `--priority`, `--custom-field key=value`, `--draft`, `--org`, `--project`

### Modified files

- `src/commands/create/issue.ts` — Replace with a thin alias that delegates to the generic command with `type=issue`
- `src/commands/create/doc.ts` — Replace with a thin alias that delegates to the generic command with `type=doc`

### Approach

1. The generic command takes the item type as the first positional arg
2. It calls the existing `createItem` gRPC method with `itemType` set to the plural form
3. Backward compatibility: `centy create issue` and `centy create doc` continue to work (either via oclif aliases or kept as thin wrappers)
4. The interactive input flow (`gatherIssueInput`) should be generalized or bypassed when flags are provided
